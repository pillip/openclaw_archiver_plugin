# UX Specification: OpenClaw Archiver Plugin

> Slack DM 기반 CLI 플러그인의 사용자 경험 명세서
> 이 문서는 사용자가 보는 모든 텍스트의 단일 출처(Single Source of Truth)이다.
> 모든 응답 메시지 템플릿은 Slack 네이티브 mrkdwn으로 작성되어 있으며, 코드 출력 시 그대로 사용해야 한다.

---

## 1. Voice & Tone

### 어조 정의

- **간결하다**: 한 문장이면 한 문장으로 끝낸다. 수식어를 붙이지 않는다.
- **정확하다**: 무엇이 되었고, 무엇을 해야 하는지 명확하게 전달한다.
- **담백하다**: 감탄사, 이모지, 느낌표를 남발하지 않는다. CLI 도구답게.

### 격식 수준

- **해요체 사용 금지**: "~했습니다", "~하세요" 대신 **간결한 명사형/평서형 종결**을 사용한다.
- 성공: `저장했습니다` (O) / `저장되었어요!` (X)
- 에러: `제목을 입력하세요.` (O) / `제목이 빠졌네요~` (X)
- 모든 응답은 **존댓말 평서형**(합쇼체/하십시오체)을 기본으로 한다.

### 우리가 쓰는 표현 vs. 쓰지 않는 표현

| 우리는 이렇게 쓴다 | 이렇게 쓰지 않는다 |
|---|---|
| `저장했습니다` | `저장 프로세스가 완료되었습니다` |
| `해당 메세지를 찾을 수 없습니다` | `null reference exception` |
| `제목을 입력하세요` | `필수 파라미터가 누락되었습니다` |
| `"Backend" 프로젝트` | `project_name=Backend` |
| `3건` | `3개의 아이템` |

## 2. Slack mrkdwn 포맷팅 규칙 (NFR-008)

### 2.1 필수 규칙

이 섹션의 규칙은 모든 응답 메시지에 예외 없이 적용된다. 구현 시 이 규칙을 위반하는 코드는 CI 테스트에서 자동으로 거부된다.

| 규칙 | 설명 | 검증 방법 |
|---|---|---|
| 코드블럭 금지 | 단일 백틱, 삼중 백틱 모두 사용하지 않는다 | 정규식: 응답 문자열에 백틱 문자 0건 |
| 8칸 들여쓰기 금지 | 8칸 이상 연속 공백으로 시작하는 줄을 만들지 않는다. Slack이 preformatted text로 렌더링한다 | 정규식: `^ {8,}` 패턴 매치 0건 |
| 4칸 들여쓰기 금지 | 4칸 이상 연속 공백으로 시작하는 줄도 만들지 않는다. 안전 마진 확보 | 정규식: `^ {4,}` 패턴 매치 0건 |
| Slack 링크 형식 필수 | 링크를 포함하는 응답에서 bare URL을 사용하지 않는다 | 모든 URL이 `<url\|text>` 형식으로 감싸져야 한다 |
| 볼드 서식 사용 | 제목, 레이블 등 강조가 필요한 텍스트에 Slack 볼드(`*text*`)를 사용한다 | 레이블에 `*` 래핑 확인 |

### 2.2 허용되는 서식 요소

| 서식 | Slack mrkdwn 문법 | 용도 |
|---|---|---|
| 볼드 | `*텍스트*` | 레이블, 헤더, 프로젝트명, 명령어 |
| 링크 | `<https://example.com\|표시 텍스트>` | 저장된 메세지 링크 |
| 줄바꿈 | `\n` | 항목 간, 섹션 간 구분 |
| 구분선 | `───` (유니코드 U+2500, 3개 이상) | 헤더와 본문 사이 시각적 구분 |
| 화살표 | `→` (U+2192) | 변경 전후 표시 |
| 일반 텍스트 | 서식 없음 | 설명, 메타 정보 |

### 2.3 금지되는 서식 요소 (NFR-008-NEG)

| 금지 패턴 | 이유 |
|---|---|
| 단일 백틱 (`` ` ``) | Slack에서 인라인 코드로 렌더링. CLI 도구 분위기와 맞지 않으며 NFR-008 위반 |
| 삼중 백틱 (`` ``` ``) | 코드블럭 렌더링. 내부 URL이 클릭 불가능해짐 |
| 4칸 이상 선행 공백 | Slack이 preformatted text로 렌더링할 수 있음 |
| bare URL | `<url\|text>` 형식으로 감싸지 않은 URL. 클릭은 가능하지만 일관성 위반 |
| Markdown 볼드 (`**text**`) | Slack mrkdwn이 아닌 Markdown 문법. Slack에서 렌더링되지 않음 |

### 2.4 제목 내 특수문자 이스케이프

사용자가 입력한 제목(title)에 `*` 문자가 포함된 경우, Slack이 의도하지 않은 볼드 렌더링을 수행할 수 있다 (R-008). 볼드 서식은 레이블에만 적용하고, 사용자 입력 텍스트(제목, 프로젝트명)는 볼드로 감싸지 않는다.

- 레이블 볼드: `*제목:* 스프린트 회의록` (O)
- 사용자 입력 볼드: `제목: *스프린트 회의록*` (X -- 제목에 `*`가 있으면 깨짐)

## 3. Information Architecture

### 명령 계층 구조

/archive 명령은 아래의 하위 명령으로 구성된다.

| 하위 명령 | 문법 | 설명 |
|---|---|---|
| save | /archive save <제목> <링크> [/p <프로젝트>] | 메세지 저장 |
| list | /archive list [/p <프로젝트>] | 목록 조회 |
| search | /archive search <키워드> [/p <프로젝트>] | 검색 |
| edit | /archive edit <ID> <새 제목> | 제목 수정 |
| remove | /archive remove <ID> | 삭제 |
| project list | /archive project list | 프로젝트 목록 |
| project rename | /archive project rename <기존이름> <새이름> | 프로젝트 이름 변경 |
| project delete | /archive project delete <프로젝트이름> | 프로젝트 삭제 |
| help | /archive help | 도움말 |

### 네비게이션 흐름

이 플러그인은 Slack DM 기반 CLI이므로 화면 전환 개념이 없다. 모든 명령은 *단일 요청 - 단일 응답* 구조이다. 사용자가 길을 잃으면 `/archive help`로 돌아올 수 있다.

## 4. Key Flows

### 4.1 메세지 저장 (`save`)

#### Happy Path -- 프로젝트 미지정

사용자 입력:

> /archive save 스프린트 회의록 https://slack.com/archives/C01/p123

봇 응답 (Slack mrkdwn):

```
저장했습니다. (ID: 7)
*제목:* 스프린트 회의록
```

#### Happy Path -- 프로젝트 지정

사용자 입력:

> /archive save 스프린트 회의록 https://slack.com/archives/C01/p123 /p Backend

봇 응답:

```
저장했습니다. (ID: 7)
*제목:* 스프린트 회의록
*프로젝트:* Backend
```

프로젝트가 처음 사용되는 경우에도 동일한 응답을 반환한다. 프로젝트 자동 생성은 내부 동작이며 사용자에게 별도로 알리지 않는다 (FR-003).

#### Error Paths

| 상황 | 응답 |
|---|---|
| 제목 누락 | 사용법: /archive save <제목> <링크> [/p <프로젝트>] |
| 링크 누락 | 사용법: /archive save <제목> <링크> [/p <프로젝트>] |
| 제목+링크 모두 누락 | 사용법: /archive save <제목> <링크> [/p <프로젝트>] |

> 필수 인자가 부족한 경우 구체적으로 무엇이 빠졌는지 구분하지 않고, 일관되게 사용법을 안내한다. 사용자가 어떤 부분을 빠뜨렸는지는 사용법을 보면 알 수 있다.

#### save 응답 문자열 템플릿 (구현용)

프로젝트 미지정 시:

```
저장했습니다. (ID: {id})\n*제목:* {title}
```

프로젝트 지정 시:

```
저장했습니다. (ID: {id})\n*제목:* {title}\n*프로젝트:* {project}
```

---

### 4.2 목록 조회 (`list`)

#### Happy Path -- 전체 조회

사용자 입력:

> /archive list

봇 응답:

```
*저장된 메세지* (3건)
───
#5 스프린트 회의록
<https://slack.com/archives/C01/p123|링크> | Backend | 2026-03-01

#3 디자인 리뷰 피드백
<https://slack.com/archives/C02/p456|링크> | 미분류 | 2026-02-28

#1 온보딩 가이드
<https://slack.com/archives/C03/p789|링크> | HR | 2026-02-25
```

#### Happy Path -- 프로젝트 필터

사용자 입력:

> /archive list /p Backend

봇 응답:

```
*저장된 메세지 — Backend* (1건)
───
#5 스프린트 회의록
<https://slack.com/archives/C01/p123|링크> | 2026-03-01
```

프로젝트별 조회에서는 프로젝트명을 각 항목에 반복하지 않는다. 헤더에서 이미 명시했기 때문이다.

#### Empty States

| 상황 | 응답 |
|---|---|
| 전체 목록이 비어있음 | 저장된 메세지가 없습니다. /archive save 로 메세지를 저장해보세요. |
| 프로젝트 내 메세지 없음 | "Backend" 프로젝트에 저장된 메세지가 없습니다. |
| 존재하지 않는 프로젝트 | "Backend" 프로젝트를 찾을 수 없습니다. |

#### list 응답 문자열 템플릿 (구현용)

헤더 -- 전체:

```
*저장된 메세지* ({count}건)
```

헤더 -- 프로젝트 필터:

```
*저장된 메세지 — {project}* ({count}건)
```

구분선 (헤더 다음 줄):

```
───
```

각 항목 -- 전체 조회 시 (프로젝트명 포함):

```
#{id} {title}\n<{link}|링크> | {project_name_or_미분류} | {date}
```

각 항목 -- 프로젝트 필터 시 (프로젝트명 생략):

```
#{id} {title}\n<{link}|링크> | {date}
```

항목 간 구분: 빈 줄 1개 (`\n\n`)

### 4.3 검색 (`search`)

#### Happy Path -- 전체 검색

사용자 입력:

> /archive search 회의록

봇 응답:

```
*검색 결과: "회의록"* (2건)
───
#5 스프린트 회의록
<https://slack.com/archives/C01/p123|링크> | Backend | 2026-03-01

#2 주간 회의록 정리
<https://slack.com/archives/C04/p012|링크> | 미분류 | 2026-02-27
```

#### Happy Path -- 프로젝트 범위 검색

사용자 입력:

> /archive search 회의록 /p Backend

봇 응답:

```
*검색 결과: "회의록" — Backend* (1건)
───
#5 스프린트 회의록
<https://slack.com/archives/C01/p123|링크> | 2026-03-01
```

#### Empty / Error States

| 상황 | 응답 |
|---|---|
| 검색 결과 없음 | "회의록"에 대한 검색 결과가 없습니다. |
| 프로젝트 범위 검색 결과 없음 | "Backend" 프로젝트에서 "회의록"에 대한 검색 결과가 없습니다. |
| 존재하지 않는 프로젝트 범위 | "Backend" 프로젝트를 찾을 수 없습니다. |
| 키워드 누락 | 사용법: /archive search <키워드> [/p <프로젝트>] |

#### search 응답 문자열 템플릿 (구현용)

헤더 -- 전체:

```
*검색 결과: "{keyword}"* ({count}건)
```

헤더 -- 프로젝트 범위:

```
*검색 결과: "{keyword}" — {project}* ({count}건)
```

항목 형식: list와 동일 (4.2절 참조)

---

### 4.4 제목 수정 (`edit`)

#### Happy Path

사용자 입력:

> /archive edit 5 스프린트 회의록 (3월)

봇 응답:

```
제목을 수정했습니다. (ID: 5)
*변경:* 스프린트 회의록 → 스프린트 회의록 (3월)
```

#### Error Paths

| 상황 | 응답 |
|---|---|
| ID에 해당하는 메세지 없음 (또는 타인 소유) | 해당 메세지를 찾을 수 없습니다. (ID: 5) |
| ID가 숫자가 아님 | ID는 숫자여야 합니다. 사용법: /archive edit <ID> <새 제목> |
| 새 제목 누락 | 사용법: /archive edit <ID> <새 제목> |
| ID 누락 | 사용법: /archive edit <ID> <새 제목> |

#### edit 응답 문자열 템플릿 (구현용)

성공:

```
제목을 수정했습니다. (ID: {id})\n*변경:* {old_title} → {new_title}
```

---

### 4.5 삭제 (`remove`)

#### Happy Path

사용자 입력:

> /archive remove 5

봇 응답:

```
삭제했습니다. (ID: 5)
*제목:* 스프린트 회의록 (3월)
```

삭제 시 제목을 함께 표시하여 사용자가 올바른 항목을 삭제했는지 확인할 수 있게 한다.

#### Error Paths

| 상황 | 응답 |
|---|---|
| ID에 해당하는 메세지 없음 (또는 타인 소유) | 해당 메세지를 찾을 수 없습니다. (ID: 5) |
| ID가 숫자가 아님 | ID는 숫자여야 합니다. 사용법: /archive remove <ID> |
| ID 누락 | 사용법: /archive remove <ID> |

> 보안 원칙 (FR-019): "존재하지 않는 메세지"와 "타인의 메세지"에 대해 동일한 에러 메시지를 사용한다. 데이터 존재 여부를 외부에서 유추할 수 없도록 한다.

#### remove 응답 문자열 템플릿 (구현용)

성공:

```
삭제했습니다. (ID: {id})\n*제목:* {title}
```

---

### 4.6 프로젝트 목록 (`project list`)

#### Happy Path

사용자 입력:

> /archive project list

봇 응답:

```
*프로젝트* (3개)
───
Backend — 5건
Frontend — 1건
HR — 2건
```

#### Empty State

```
프로젝트가 없습니다. /archive save <제목> <링크> /p <프로젝트> 로 메세지를 저장하면 프로젝트가 자동으로 생성됩니다.
```

#### project list 응답 문자열 템플릿 (구현용)

헤더:

```
*프로젝트* ({count}개)
```

구분선:

```
───
```

각 항목:

```
{name} — {archive_count}건
```

항목 간 구분: 줄바꿈 1개 (`\n`)

### 4.7 프로젝트 이름 변경 (`project rename`)

#### Happy Path

사용자 입력:

> /archive project rename BE Backend

봇 응답:

```
프로젝트 이름을 변경했습니다.
*변경:* BE → Backend
```

#### Error Paths

| 상황 | 응답 |
|---|---|
| 기존 프로젝트 없음 | "BE" 프로젝트를 찾을 수 없습니다. |
| 새 이름이 이미 존재 | "Backend" 프로젝트가 이미 존재합니다. 다른 이름을 입력하세요. |
| 인자 부족 | 사용법: /archive project rename <기존이름> <새이름> |

#### project rename 응답 문자열 템플릿 (구현용)

성공:

```
프로젝트 이름을 변경했습니다.\n*변경:* {old} → {new}
```

### 4.8 프로젝트 삭제 (`project delete`)

#### Happy Path -- 메세지가 있는 프로젝트

사용자 입력:

> /archive project delete Backend

봇 응답:

```
"Backend" 프로젝트를 삭제했습니다.
5건의 메세지가 미분류로 변경되었습니다.
```

#### Happy Path -- 메세지가 없는 프로젝트

봇 응답:

```
"Backend" 프로젝트를 삭제했습니다.
```

미분류 변경 안내는 실제로 영향받은 메세지가 있을 때만 표시한다.

#### Error Paths

| 상황 | 응답 |
|---|---|
| 프로젝트 없음 | "Backend" 프로젝트를 찾을 수 없습니다. |

## 6. Copy Guidelines

### 6.1 메시지 패턴 공식

#### 성공 메시지 공식

```
[무엇을 했는지] + [확인 정보 (볼드 레이블)]
```

예: `저장했습니다. (ID: 7)` + `\n*제목:* 스프린트 회의록`

#### 에러 메시지 공식

```
[무슨 상황인지] + [어떻게 하면 되는지]
```

예: `"Backend" 프로젝트가 이미 존재합니다.` + `다른 이름을 입력하세요.`

#### 빈 상태 공식

```
[현재 상태] + [다음 행동 안내]
```

예: `저장된 메세지가 없습니다.` + `/archive save 로 메세지를 저장해보세요.`

#### 인자 부족 에러 공식

사용법을 바로 보여준다. 불필요한 설명을 앞에 붙이지 않는다.

```
사용법: /archive {command} {arguments}
```

### 6.2 목록/검색 출력 포맷 (mrkdwn)

#### 구조

```
*{헤더}* ({N}건)
───
#{id} {title}
<{link}|링크> | {project_or_미분류} | {date}

#{id} {title}
<{link}|링크> | {project_or_미분류} | {date}
```

- 헤더: 볼드(`*...*`)로 감싼다. 총 건수를 괄호로 표시.
- 구분선: `───` (U+2500 3개). 헤더 바로 아래 별도 줄에 배치.
- 각 항목 1행: `#{id} {title}` -- ID와 제목. 들여쓰기 없이 줄 시작.
- 각 항목 2행: `<{link}|링크> | {메타정보}` -- Slack 링크 형식 + 프로젝트/날짜 메타.
- 항목 간 구분: 빈 줄 1개.
- 날짜 형식: `YYYY-MM-DD`
- 프로젝트와 날짜 사이에 ` | ` 구분자 사용.
- 프로젝트가 없으면 `미분류` 표시.

#### 프로젝트 필터 시

프로젝트명은 헤더에만 표시하고 각 항목에서는 생략한다.

```
#{id} {title}
<{link}|링크> | {date}
```

#### 프로젝트 목록 출력 형식

```
*프로젝트* ({N}개)
───
{name} — {count}건
{name} — {count}건
```

- 프로젝트 수는 `개`, 메세지 수는 `건`으로 단위를 구분한다.
- 프로젝트명과 건수 사이에 ` — ` (em dash) 구분자 사용.

#### 변경 사항 표시

변경 전후를 보여줄 때는 볼드 레이블과 화살표 `→` 를 사용한다.

```
*변경:* BE → Backend
*변경:* 스프린트 회의록 → 스프린트 회의록 (3월)
```

### 6.3 용어 사전 (Glossary)

모든 응답에서 아래 용어를 일관되게 사용한다. 동의어를 혼용하지 않는다.

| 정규 용어 | 사용하지 않는 표현 | 설명 |
|---|---|---|
| 메세지 | 아카이브, 항목, 아이템, 레코드 | 사용자가 저장한 링크 단위 |
| 제목 | 타이틀, title, 이름 | 메세지의 설명 텍스트 |
| 프로젝트 | 그룹, 폴더, 카테고리, 태그 | 메세지 분류 단위 |
| 미분류 | 없음, 기본, default, 미지정 | 프로젝트가 지정되지 않은 상태 |
| 저장 | 아카이빙, 북마크, 등록 | save 동작 |
| 삭제 | 제거, 해제 | remove/delete 동작 |
| 수정 | 편집, 변경, 업데이트 | edit 동작 |
| 검색 | 찾기, 조회 | search 동작 |
| 목록 | 리스트, list | list 동작 |
| 건 | 개 (메세지에 한정) | 메세지 수의 단위 |
| 개 | 건 (프로젝트에 한정) | 프로젝트 수의 단위 |

### 6.4 숫자/날짜 표기 규칙

| 항목 | 형식 | 예시 |
|---|---|---|
| 날짜 | `YYYY-MM-DD` | `2026-03-01` |
| 메세지 ID (목록 내) | `#` + 숫자 | `#5` |
| 메세지 ID (단건 응답) | `ID: ` + 숫자, 괄호로 감싸기 | `(ID: 5)` |
| 메세지 건수 | 숫자 + `건` | `3건` |
| 프로젝트 수 | 숫자 + `개` | `2개` |

### 6.5 금지 사항

- 개발 용어를 사용자에게 노출하지 않는다: `null`, `invalid`, `exception`, `error code`, `parameter`, `422` 등
- 느낌표를 에러 메시지에 사용하지 않는다
- 이모지를 사용하지 않는다 (Slack 환경이지만, CLI 도구로서 담백함을 유지)
- 같은 동작을 다른 단어로 부르지 않는다 (용어 사전 참조)
- "오류가 발생했습니다" 같은 모호한 에러 메시지를 사용하지 않는다
- 수동태를 에러 메시지에 사용하지 않는다: `찾을 수 없습니다` (O) / `찾아지지 않았습니다` (X)
- 코드블럭(백틱)을 응답 메시지에 사용하지 않는다 (NFR-008)
- 4칸 이상의 선행 공백을 응답 메시지에 사용하지 않는다 (NFR-008)

## 7. Edge Case 처리

### 7.1 제목에 공백이 포함된 경우

`save` 명령에서 URL 패턴(`https://...`)을 먼저 추출하고, 나머지 텍스트를 제목으로 처리한다. `/p` 옵션이 있으면 그것도 먼저 분리한다 (R-002 대응).

```
입력:  /archive save 3월 스프린트 회의록 https://slack.com/archives/C01/p123 /p Backend
파싱:  title = "3월 스프린트 회의록", link = "https://...", project = "Backend"
```

### 7.2 제목 텍스트에 `/p`가 포함된 경우

`/p`는 명령 문자열의 *마지막에 위치한 것만* 옵션으로 인식한다. 제목 중간의 `/p`는 제목의 일부로 처리한다 (R-001 대응).

```
입력:  /archive save a/p 패턴 분석 https://slack.com/archives/C01/p123
파싱:  title = "a/p 패턴 분석", link = "https://...", project = null
```

```
입력:  /archive save a/p 패턴 분석 https://slack.com/archives/C01/p123 /p Backend
파싱:  title = "a/p 패턴 분석", link = "https://...", project = "Backend"
```

### 7.3 존재하지 않는 프로젝트로 list/search

`list`와 `search`에서 `/p` 옵션에 존재하지 않는 프로젝트를 지정하면, 빈 결과가 아닌 *프로젝트 자체를 찾을 수 없다는 에러*를 반환한다. 이는 오타를 빠르게 인지할 수 있게 한다.

```
봇: "Baceknd" 프로젝트를 찾을 수 없습니다.
```

### 7.4 타인의 데이터 접근 시도

`edit`, `remove`에서 타인의 메세지 ID를 사용하면, "존재하지 않는 ID"와 동일한 에러를 반환한다. 존재 여부를 유추할 수 없도록 한다 (FR-019).

```
봇: 해당 메세지를 찾을 수 없습니다. (ID: 99)
```

### 7.5 프로젝트 관리에서 타인의 프로젝트

`project rename`, `project delete`에서 타인의 프로젝트명을 사용하면, 존재하지 않는 프로젝트와 동일한 에러를 반환한다 (FR-019).

```
봇: "SecretProject" 프로젝트를 찾을 수 없습니다.
```

### 7.6 제목에 Slack mrkdwn 특수문자가 포함된 경우

사용자가 입력한 제목에 `*`, `~`, `_` 등 Slack mrkdwn 서식 문자가 포함된 경우, 의도하지 않은 렌더링이 발생할 수 있다 (R-008). 현재 버전에서는 이스케이프를 수행하지 않지만, 볼드 서식을 사용자 입력이 아닌 레이블에만 적용하여 영향을 최소화한다.

```
입력 제목: "v1*beta* 릴리즈 노트"
표시: *제목:* v1*beta* 릴리즈 노트
-- "beta"가 의도치 않게 볼드로 렌더링될 수 있음
```

이 문제는 v0 범위에서 허용하며, 빈도가 높아지면 이스케이프 로직 도입을 검토한다.

### 7.7 Slack 메시지 길이 제한

Slack 메시지 최대 길이는 4,000자이다. 목록이 길어질 경우를 대비하여, 항목당 약 80자(mrkdwn 서식 포함)로 계산하면 약 40건까지 한 메시지에 표시 가능하다.

현재 버전에서는 페이지네이션이 범위 외이므로, 결과가 많을 경우 Slack 메시지 길이 제한에 의해 잘릴 수 있다. 이 경우 `search`나 `/p` 옵션으로 범위를 좁혀 조회하도록 안내한다.

## 8. Accessibility Notes

### 8.1 Slack DM 환경에서의 가독성

- *코드 블록 사용 금지*: NFR-008에 따라 모든 응답에서 코드 블록을 사용하지 않는다. Slack mrkdwn 서식(볼드, 링크)으로 가독성을 확보한다.
- *링크 클릭 가능성*: 모든 URL은 `<url|링크>` Slack 링크 형식으로 표시되어 클릭 가능한 하이퍼링크로 렌더링된다. 코드 블록 미사용으로 이 문제가 근본적으로 해소되었다.
- *시각적 구분*: 유니코드 구분선(`───`)과 빈 줄로 항목 간 구조를 표현한다.

### 8.2 스크린 리더 호환성

- 유니코드 장식 문자(구분선 `─`)는 스크린 리더에서 "가로선" 등으로 읽힐 수 있다. 구분선은 순수하게 시각적 구분 용도이며, 정보를 담지 않는다.
- 중요한 정보(ID, 제목, 링크)는 구분선이나 장식 없이 텍스트로만 전달되도록 한다.
- 목록의 각 항목은 `#{id}` 로 시작하여 순서를 인지할 수 있게 한다.

- [ ] 사용자 입력 텍스트(title, project name)는 볼드로 감싸지 않음

### 9.3 용어/표기 일관성

- [ ] 날짜 형식이 `YYYY-MM-DD`
- [ ] 메세지 수 단위 `건`, 프로젝트 수 단위 `개` 일관 사용
- [ ] 용어 사전의 정규 용어만 사용
- [ ] 변경 전후 표시에 `→` 화살표 사용
- [ ] 구분선에 `───` (U+2500) 사용

### 9.4 자동 검증 (CI용)

다음 정규식 패턴으로 모든 cmd_*.py 핸들러의 반환 문자열을 검증한다:

| 검증 항목 | 패턴 | 기대 매치 수 |
|---|---|---|
| 8칸 들여쓰기 없음 | `^ {8,}` (각 줄 대상) | 0 |
| 4칸 들여쓰기 없음 | `^ {4,}` (각 줄 대상) | 0 |
| 백틱 없음 | `` ` `` | 0 |
| bare URL 없음 (link 포함 응답만) | `(?<![\|<])https?://\S+(?![\|>])` | 0 |
| Slack 링크 형식 사용 | `<https?://[^\|]+\|[^>]+>` | 1 이상 (link 포함 응답만) |
